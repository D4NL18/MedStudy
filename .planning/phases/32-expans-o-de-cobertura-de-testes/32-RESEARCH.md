# Phase 32: Expansão de Cobertura de Testes - Research & Planning Guide

## Understanding the Scope
This phase focuses on significantly improving test coverage, specifically targeting:
1. **Backend Edge Cases (TEST-01):** Validating behavior under HTTP 400 (Bad Request), 403 (Forbidden), 404 (Not Found), and 429 (Too Many Requests).
2. **Frontend UI & Complex Services (TEST-02):** Hybrid UI testing (isolated vs. deep integration) and service testing (e.g., Spaced Repetition logic).

## Backend Strategy: Edge Cases

### 1. Exception Handling Architecture
The backend uses a `@RestControllerAdvice` named `GlobalExceptionHandler` to handle exceptions centrally. 
- **400 Bad Request:** Handled via `MethodArgumentNotValidException` (triggered by `@Valid` on DTOs).
- **403 Forbidden:** Handled via `AccessDeniedException` (triggered by Spring Security `@PreAuthorize` or missing permissions).
- **404 Not Found:** Handled via `EntityNotFoundException`.
- **429 Too Many Requests:** Handled by a custom `HandlerInterceptor` (`RateLimitInterceptor`) that uses Bucket4j.

### 2. Testing Approaches
According to `32-CONTEXT.md`, the backend tests must use a dual strategy:
- **Unit/WebMvcTests:** `@WebMvcTest` with mocked services (`@MockBean`) to quickly test Controller validation (400) and authorization (403).
- **Integration Tests:** `@SpringBootTest` paired with H2 (in-memory) to ensure database constraints and actual Spring Security filters work correctly, particularly for 404s and rate limiting.

### 3. Rate Limit Testing (429) Constraints
The `RateLimitInterceptor` currently hardcodes limits (e.g., 100/min for users, 7/min for anonymous). 
- **Refactoring Required:** Before writing the rate limit tests, `RateLimitInterceptor` needs to be updated to read capacities from application properties (e.g., `@Value("${rate.limit.user:100}")`) so the test profile can override it.
- **Test Implementation:** The context states the limit must be set to **20 requests/minute** in the test profile. The test will perform a loop of 20 successful requests, and assert that the 21st returns HTTP 429.

## Frontend Strategy: UI & Services

### 1. Testing Frameworks
The project uses Angular 21 with Karma, Jasmine, and `ng-mocks`.

### 2. Hybrid UI Testing Approach
As defined in the context, UI components will be tested using two patterns:
- **Isolated (Shallow) Tests:** Use `MockBuilder` from `ng-mocks` or `schemas: [NO_ERRORS_SCHEMA]` to mock child components and focus purely on the component's internal logic and template bindings.
- **Deep Integration Tests:** For container/page components (e.g., `FlashcardsStudyComponent` in `src/app/features/flashcards/pages/flashcards-study`), render the full DOM to ensure child components interact correctly with the parent.

### 3. Complex Services (Spaced Repetition)
The Spaced Repetition logic is driven by components interacting with `FlashcardService` and `RevisionService`. Testing must mock HTTP calls (`HttpTestingController`) and validate:
- Calculation and submission of ratings (`rateFlashcard`).
- State management during the study queue (advancing to the next card, finishing the session).

## Validation Architecture

*(Required for the Nyquist validation step)*

To successfully validate this phase, the following architectural tests and checks must be implemented and pass:

### Backend Validation (Edge Cases)
1. **HTTP 400 Validation:** 
   - Ensure a `@WebMvcTest` sends an invalid payload (e.g., missing required fields) to a POST/PUT endpoint.
   - **Assertion:** Expect status `400 Bad Request` and check that the response body contains the specific validation error message generated by `GlobalExceptionHandler`.
2. **HTTP 403 Validation:** 
   - Ensure a test attempts to access an endpoint requiring a specific role/owner with an unauthorized JWT token or a generic user.
   - **Assertion:** Expect status `403 Forbidden` and the "Acesso Negado" message.
3. **HTTP 404 Validation:** 
   - Ensure a `@SpringBootTest` attempts to fetch a resource with a non-existent UUID or ID.
   - **Assertion:** Expect status `404 Not Found` handled by `EntityNotFoundException`.
4. **HTTP 429 Validation:** 
   - Override the rate limit capacity to 20 in the test configuration properties.
   - Loop 20 requests and assert they return 2xx.
   - Fire the 21st request.
   - **Assertion:** Expect status `429 Too Many Requests`.

### Frontend Validation (UI & Services)
1. **Shallow UI Tests:**
   - Verify that presentational components render correctly based on `@Input()` data without rendering child dependencies.
2. **Deep UI Integration Tests:**
   - Render a primary page (e.g., `FlashcardsStudyComponent`).
   - Trigger a user action (e.g., clicking a rating button like "Fácil" ou "Difícil").
   - **Assertion:** Verify that the correct service method is called with the expected arguments, and the UI transitions to the next card or completion state.
3. **Complex Service Tests (Spaced Repetition):**
   - Inject `FlashcardService` alongside `HttpTestingController`.
   - **Assertion:** Ensure that calls to `rateFlashcard` serialize the payload correctly and handle the observable response as expected. 

### Coverage Thresholds
- The backend `pom.xml` requires **80% line coverage** (`jacoco-maven-plugin`). Validation must include running `mvn clean test jacoco:report jacoco:check` and ensuring it passes.
- Frontend validation should include `ng test --no-watch --code-coverage` to ensure tests execute successfully.
